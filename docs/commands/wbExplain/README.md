# wbExplain — ELI5

### What does it do?
Imagine you are playing with LEGOs and someone hands you instructions that are too complicated. `/wbExplain` is like asking an older sibling, "Hey, how does this piece connect to that piece?" instead of asking them to actually build it for you. 

It generates a clear, readable explanation of a task or a part of the codebase and saves it so everyone can read it later.

### When to use it?
- Before running `/wbWork`, if you don't understand the task.
- When you encounter a complex part of the system and want a simplified breakdown.
- When you want to document a complex concept in multiple languages or styles.

### How to use it?
Run it on a plan file with a specific task ID:
`/wbExplain plan.md --id=2 --as=eli5`

Or ask it a question about a folder:
`/wbExplain packages/wb-core/ "How does authentication work here?"`

# wbExplain — Expert

### Architectural Purpose
`/wbExplain` acts as the primary "Knowledge Distillation" layer in the WB-Labs monorepo architecture. 

It explicitly separates state-mutating execution logic (`/wbWork`, `/wbRefactor`) from state-querying pedagogical logic. By routing queries through this command, we enforce the generation of versioned, structurally formatted Tier-1 markdown artifacts (with 4D Temporal Navigation footers) rather than ephemeral chat interactions.

### Execution Flow
1. **Target parsing**: Identifies if the target is a `.md` plan file (task parsing) or a directory (sub-system analysis).
2. **Intent mapping**: Maps the user's `--as=<style>` flag strictly against the `shortcuts.md` configuration to inherit systemic personas and translation logic.
3. **Smart Merge**: If a daily explanation file already exists (`explain_<slug>_<YYYYMMDD>.md`), it safely appends the new response to prevent namespace fragmentation.
4. **Output formatting**: Emits a three-part structure (Summary, Deep Dive, Recommended Approach) and auto-injects the Tier 1 global footer.

### Advanced Usage
Combine with other commands for pipeline operations. For example, when an audit fails via `/wbReview`, pipe the failure ID into `/wbExplain --as=expert` to generate a root-cause hypothesis document before dispatching the fix via `/wbWork`.

# wbExplain — Practical

### The Day-to-Day Workflow

When working in the monorepo, you will often encounter tasks generated by `/wbPlan` that are ambiguous or complex. 

Instead of jumping straight into `/wbWork` and risking a bad implementation, use `/wbExplain`.

### Step 1: Generate the Explanation
Run the command against the plan file containing the task:
```bash
/wbExplain packages/wb-core/.wb/workflows/reports/2026/05/03/plans/plan_wb-core_20260503.md --id=1
```

### Step 2: Review the Output
Open the newly generated artifact located in the `explanations/` or `tasks/task_<N>/` folder. It will look like this:
- **High-Level Summary:** A quick synopsis of what the task means.
- **Deep Dive:** A breakdown of the code involved and the exact problem.
- **Recommended Approach:** A structured list of steps to resolve the task.

### Step 3: Execute
Once you have read the explanation and agree with the recommended approach, you can confidently run the mutation command:
```bash
/wbWork packages/wb-core/ --id=1
```
